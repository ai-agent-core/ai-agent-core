# Secrets management

Use this skill **whenever a secret is being introduced, rotated,
or audited** — API keys, tokens, signing keys, DB passwords, TLS
private keys, OAuth client secrets, webhook secrets, anything
that grants authority.

Authoritative source: `rules/SECRETS_RULES.md` and
`principles/SECURITY_PRINCIPLES.md`.

---

## Premise

A secret is a capability. Anyone with it can do what the secret
authorizes. Treat its lifecycle — generation, distribution, use,
rotation, revocation — with the same rigor as the database it
protects.

---

## Step 1 — Inventory

For every secret the system touches, know:

- **name** following the convention `<env>/<service>/<purpose>`,
- **owner** (person or team),
- **purpose** (what it grants),
- **scope** (which workload reads it),
- **rotation cadence**,
- **last rotation date**,
- **next rotation date**.

A secret without an owner is a future incident.

---

## Step 2 — Source of truth

Production secrets live in a **secrets manager**:

- AWS Secrets Manager / SSM Parameter Store,
- Google Secret Manager,
- Azure Key Vault,
- HashiCorp Vault,
- 1Password / SOPS / SealedSecrets for some workflows.

Workloads read secrets at boot or via a CSI driver.

Forbidden:

- secrets in source control (git history is forever),
- secrets in build artifacts (Dockerfile, image layers),
- secrets in CI logs,
- shared secrets across environments,
- copying production secrets to laptops.

---

## Step 3 — Distribution

For workloads:

- prefer **platform-native identity** to read secrets — no
  static credential to read other credentials,
- IAM roles for service accounts (AWS),
- workload identity (GCP / GKE),
- managed identity (Azure),
- OIDC federation for CI to cloud (no long-lived keys in CI),
- mTLS / SPIFFE for service-to-service inside the mesh.

For humans:

- SSO / IdP with MFA,
- per-secret access policy,
- time-bounded elevation,
- access logged and reviewable.

Forbidden:

- shared admin accounts to the secrets manager,
- ambient credentials authorized for production,
- developers reading prod secrets to "debug locally."

---

## Step 4 — Generation

- random material from a CSPRNG (`crypto/rand`, `secrets`,
  `getrandom`); never `Math.random()`,
- entropy ≥128 bits for tokens / session IDs,
- entropy ≥256 bits for long-lived signing keys,
- generated where they will be used; transit minimized.

For asymmetric key pairs:

- generated by KMS / HSM where possible — private key never
  leaves the trust boundary,
- versioned for rotation,
- public key distributed; private key access logged.

For symmetric keys:

- envelope encryption with KMS preferred over raw symmetric
  keys,
- versioned (`kid`) so rotation is forward-compatible.

---

## Step 5 — Application discipline

In code:

- secrets loaded once at boot (or refreshed on schedule);
  never embedded in source,
- wrapped in types that prevent accidental logging
  (`Secret<T>` with redacted `Display` / `toString`),
- comparison via constant-time equality library API,
- errors do not leak secrets in messages,
- backtraces / panic dumps do not include secrets (configure
  framework to redact).

Forbidden:

- logging the request body of authentication endpoints,
- printing environment variables in startup banners,
- sending secrets in error responses,
- secrets passed via command-line arguments (visible in
  process listings).

---

## Step 6 — Rotation

Every secret has a documented rotation schedule. Defaults:

- short-lived OIDC / IdP tokens: minutes-to-hours (issued
  automatically),
- API keys / passwords: 90 days unless platform supports shorter,
- session signing secrets: ≥annually, with multi-key support
  during rotation,
- TLS certs: automatic via ACME / ACM / cert-manager.

Rotation is automated where the platform supports it. Manual
rotation is a process, not a hope:

- next rotation date known,
- procedure documented,
- performed at least once.

A rotation that has never been done will fail the day it is
needed.

### Multi-version rotation

For session signing keys, JWT verification keys, encryption
keys:

- support **multiple active versions** during the rotation
  window,
- producers move to the new version first,
- consumers accept old + new for the window,
- old is retired after the window.

Forbidden: "switch all at once" rotations that cause downtime.

---

## Step 7 — Compromise response

Treat any of the following as compromise:

- secret committed to git (even if the commit is reverted),
- secret printed in a log,
- secret displayed on a screen recorded externally,
- secret shared in chat / email / ticket,
- secret found by a scanner (in source / in logs / on disk).

On compromise:

1. **Rotate immediately.**
2. **Audit usage during the exposure window** (was anything
   accessed that should not have been?).
3. **Investigate root cause**; fix the leak class, not just the
   symptom.
4. **Detection improvement** — make this class of leak get
   caught earlier next time.
5. Reverting the commit is not enough. The secret is on the
   attacker's laptop now.

If a secret has been in git history, the secret is already
compromised even if the commit is reverted; rotate first.

---

## Step 8 — Detection

CI / pre-commit MUST scan for secrets:

- `trufflehog`, `gitleaks`, `detect-secrets`, or
  platform-native scanner,
- block PRs that introduce secrets,
- alert on secrets that slip through to `main` or to logs.

Production logs MUST be scanned for secret patterns:

- alerts on detected patterns,
- redaction at the source,
- forensic process when a secret is found in logs.

---

## Step 9 — CI / CD secrets

CI is itself an attack target:

- prefer **OIDC federation** to cloud IAM over long-lived static
  keys,
- scope per workflow / per environment,
- mask in logs,
- third-party CI actions / images pinned by SHA,
- forbid actions that need broad privileges without review.

A compromised CI is a compromised production. Pretend it will
happen and design accordingly.

---

## Step 10 — Webhook secrets

Receiving:

- shared secret per partner / integration, never global,
- HMAC over body and timestamp,
- timestamp window enforced,
- signature verified before any processing.

Sending:

- signing secret per consumer,
- documented signature scheme,
- rotation procedure known to consumer.

---

## Step 11 — Multi-tenant secrets

When the system holds secrets *belonging to its tenants* (e.g.
their integration API keys):

- store encrypted per tenant,
- envelope encryption with KMS-protected keys,
- access checks include tenant ownership,
- right-to-deletion includes secret destruction,
- usage logs are tenant-scoped.

A tenant secret is a special category of liability.

---

## Step 12 — Verify

Periodically (e.g. quarterly):

- enumerate all secrets in the secrets manager and confirm each
  has an owner and an active workload,
- attempt a rotation on a non-critical secret,
- attempt to read a secret with mismatched workload identity →
  rejected,
- run a secret scanner on git history,
- confirm CI logs are clean of secrets.

---

## Forbidden

- "just put it in `.env` for now,"
- one secret used across multiple environments,
- secrets passed via command-line arguments,
- secrets bundled into container images,
- logging the very secret you are loading,
- rotation policy that no one knows about,
- personal credentials shared with the team,
- passwords printed to stdout in helper scripts.

---

## When this skill says STOP

- a secret enters the system without an owner → assign before
  use,
- rotation has never been performed → schedule and rehearse,
- a secret has been seen in logs / git → rotate immediately
  before continuing.

Treat every secret as if it will leak someday. The leak you can
recover from is the one you detected, rotated, and audited
immediately.

If a secret has never been rotated, it has not been managed.
