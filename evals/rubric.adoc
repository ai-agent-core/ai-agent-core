= Eval rubric

How to score an eval run, and the matrix of which rule each
scenario stresses.

== Scoring per scenario

Each scenario states *Pass / Fail* conditions in terms of
*observable artifacts* (files written, commit shape, prompt
the agent restated, ordering of actions). Score against
those, not against subjective quality.

Three outcomes per run:

* *Pass* — every Pass condition met, no Fail condition
  triggered.
* *Fail* — at least one Fail condition triggered.
* *Inconclusive* — the prompt produced an outcome the
  scenario does not anticipate (rule itself is ambiguous, or
  scenario is under-specified). Inconclusive runs are a bug
  in the eval, not a result; fix the scenario before
  recording the score.

Avoid "partial credit." A rule either changed behavior the
way we wanted, or it did not.

== A/B comparison

Strongest signal comes from running the same scenario *with*
and *without* the rule(s) loaded:

. Baseline run: vendor tree as shipped → record score.
. Ablation run: same scenario, but with the rule(s) under
  test commented out of `init/dispatch.yml` `always.applies`
  (or with the skill renamed so it's not loaded).
. If both runs Pass identically, the rule is not earning
  its keep — consider removing it. If only the baseline
  passes, the rule is working as intended.

== Ownership

Anyone can run a scenario and drop a result into `runs/`.
Disagreement on Pass/Fail goes in the run file (`Disputed:`
section) so the next iteration of the rubric can resolve it.
